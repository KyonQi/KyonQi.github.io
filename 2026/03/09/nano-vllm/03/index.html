【nano-vllm 学习】03 - 自顶向下的引擎调度层源码学习1

立志写出让纯夏都能看懂的教程讲解

我们依然将按照“自顶向下”的顺序逐层解析 nano-vllm 的实现细节。进入到引擎调度层后，代码开始稍微有些复杂了，因此这一层我打算做多篇来写，也方便各位阅读理解。在本节，我们将重点解析 llm_engine.py 以及 sequence.py 的主要思路，并辅助以部分调度器 scheduler.py 的代码。

我们在本节的目标很简单：从整体上了解引擎调度和执行工作的设计思路，对于细节的实现暂且放在下回分解。

在上篇文章中，我们抽象出了用户接口层应当实现的 4 个步骤，并具体解析了其中的第 1 和 4 点，我们将在这篇文章对其中的2、3点解析补完。具体而言，我们的思路是这样的：

prompt 请求输入 → 请求入队 → 循环：1. 单步 step 推理, 2. 存储输出 {seq_id : token_ids} → 完成推理，输出 output。

请求入队

prompt 传入的时候，因为是 Python str，需要使用 tokenizer 将 str 转化为 token 索引 List[int] 。

我们很清楚，一个 token 索引列表就是一个需要完成的推理任务，但为了更清楚简洁的表现这个任务，我们选择把所有与这个“任务”有关的信息封装进 Sequence 对象里，比如：token 索引列表、任务状态、预测token需要的采样参数 SamplingParams 等等。

😤

毕竟，你也不想后面函数每次执行推理的时候都要拖家带口地把 token_ids, sampling_params 等一系列上下文变量传来传去吧。

💡

从现在开始，所有的 prompt 推理任务都是以 Sequence 为单位进行的的，一个 Sequence 就是一个推理任务。

因此，我们的任务入队，就是将封装好的 Sequence 作为任务加入到调度器的等候任务队列。

单步 Step 驱动推理

推理的 prefill 和 decode 阶段

在具体到代码之前，我们首先做一些在 LLM 推理过程中的简单概念介绍。

对于 LLM 的具体推理过程，我们一般会分成两个阶段：

  • prefill 阶段：在这个阶段，模型处理所有初始输入的 prompt 序列，并输出首个预测的 token。这个阶段会把输入据序列的 K, V 通过 attention 计算出来，并全部存入显存中，是计算密集型。
  • decode 阶段：在 prefill 后，每步的输入只是上一步的 token，输出只生成这一个 token 对应的 logits 预测。为了避免重复计算，模型会使用 KV Cache：即把 prefill 阶段和先前 decode 阶段对应 token 的 K, V 向量累积存入显存中。如此一来，在 decode 阶段，新 token 时需要计算对应的 Q 向量，并与历史的 K, V 矩阵做 attention 就行了，生成的新的 K, V 会追加写入到 KV Cache 中。毫无疑问，这个阶段需要频繁从显存中读取 KV Cache 是内存密集型。

代码解析

这里是整个推理引擎框架的核心了，在具体讲解之前，我们同样先主动思考下：

已知所有需要执行的任务 Sequence 都已经在调度器的任务队列里了，现在组织希望你能完成对任务的推理工作设计，你会如何设计拆解整个工作流？

我吗？

如果是我的话：

  1. 调度器在任务队列里取出任务；
  2. 把任务丢给模型做 forward 计算；
  3. 整理预测的 token，把结果返回。

是的，没错，实际上也就是这么的朴实无华。

首先，对于第一个“调度取出任务”这个工作，作者正是通过 scheduler.schedule() 得到本轮需要处理的所有 sequences 实现的(注意取出的是一个 batch， List[Sequence] )。我们先简单剧透下，当继续深入 Scheduler 去剖析调度的具体实现时，我们可以很轻松的发现：调度返回的这个列表是严格区分 prefill 和 decode 两个阶段的 —— 换句话说，在当前这一轮需要处理的 Sequence 列表里，要么全是需要 prefill 的，要么全是需要 decode 的，不会两者混杂 (两个阶段计算方法并不相同)；

对于我们预想的第二步，当拿到需要处理的 Sequence 列表后，engine 会进一步传递给 ModelRunner 来真正执行模型推理计算，ModelRunner 会把计算的结果，也就是预测的下一个 token 作为输出结果返回回来 ( List[Int] 里面每个元素按照位置对应每个 Sequence 的下一个 token)。那么 ModelRunner 到底是怎么做的计算呢？且容我做点悬念，我们姑且置之待下回分解，这里大家有个概念：推理结果是从 ModelRunner 算出来的即可。

对于第三步，当拿到了各个 Sequence 的预测 token 后，当然是需要把这些 token 拼回对应的 Sequence 中，这一步是由 Scheduler.postprocess() 完成的。同时，会有一些判断结束的细节需要处理；

每一轮 step()都会向外返回已经完成推理的 Sequence，每一轮 step() 中还会统计本轮参与推理的 Sequence 总共的 token 数量，并在后续以正负的形式辅助 engine 区分 prefill 和 decode 两个阶段的 token 数据 (比如看到负的 token 总数就知道是来自 decode 阶段)。

Scheduler 和 ModelRunner 的区别？

在 step() 的实现中，我们终于开始接触调度器 scheduler 和 执行器 model_runner 了，为了避免对这两功能的混淆，我们这里简单做个功能总结：

  • Scheduler 只负责调度任务，输出本轮需要处理的 sequence 列表；
  • ModelRunner 只负责执行任务，输入需要处理的 sequence 列表，输出对应的预测 tokens。

输出 output token

在完成所有的 Sequence 任务后，我们还需要把以 List[int] 形式存在的 token_ids 重新转换为我们人能看的 Python str，毕竟从拿来到哪去，自然需要 tokenizer 的解码帮助，在完成解码后将文本结果正常返回给用户。
